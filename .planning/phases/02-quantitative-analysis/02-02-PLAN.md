**Execution Steps — implement as a single step containing the 3-stage agent loop, not as 3 separate steps:**

Step 1 — Run the invisible agent loop (Stage 1 → Stage 2 → Stage 3). No other steps needed; the agent loop IS the execution.

**Stage 1 — Statistician (internal, never shown to user):**

Six sub-steps to include in the prompt instructions:

1. Determine data type of VarX and VarY (categorical 2-group, categorical 3+ group, continuous). Read the actual data to determine types — do not assume.
2. Consult the Statistical Test Selector Matrix in `agents/agent-statistician.md` for the recommended test. The matrix is a GUIDE, not a rigid rule (per D-05) — the Statistician exercises judgment based on observed data characteristics.
3. Check test assumptions: sample size per group (N ≥ 30 for parametric per constitution.md Article 5), normality (if t-test/ANOVA), homogeneity of variance (Levene's test), expected cell frequencies ≥ 5 for χ² (fall back to Fisher's exact if violated).
4. If parametric assumptions are violated, fall back to non-parametric equivalent: Mann-Whitney U for t-test, Kruskal-Wallis for ANOVA, Spearman's ρ for Pearson's r.
5. Compute the test statistic, degrees of freedom, p-value, and effect size (Cohen's d for mean comparisons, Cramér's V for χ², η² for ANOVA, r² for correlation).
6. Compute all percentages with margin of error per constitution.md Article 1: MoE = 1.96 × √(p(1-p)/n).

**Stage 2 — Critic (internal, never shown to user):**

Four validation checks to include in the prompt instructions:

1. Validate test selection: is this the right test for the observed data types? Are assumptions met? Did the Statistician check them?
2. Verify all constitution.md articles: Article 1 (MoE on every %), Article 2 (p < 0.05 for significance claims), Article 5 (parametric test N ≥ 30 per group), Article 6b (% sums within 99.5%-100.5%).
3. Flag issues: small sample sizes (flag per constitution), spurious correlations (no plausible causal mechanism), overgeneralization risks (convenience sample, single segment).
4. If test selection is invalid OR constitution articles are violated → return to Statistician with specific correction. Do NOT pass invalid output to Tufte Designer.

**Stage 3 — Tufte Designer (ONLY output user sees):**

Four formatting instructions:

1. Format as dense crosstab table. The header row MUST contain the key finding (conclusion-first). The table has EXACTLY these columns: `{VarX}` (left-aligned), `N` (center-aligned), `{VarY Category A}` (center-aligned), `{VarY Category B}` (center-aligned), `Teste` (left-aligned). Add categories dynamically for 3+ group VarY — not just 2 columns.
2. Every row: segment name, N for that row, count and % with MoE for each VarY category. Total row at bottom with overall N and test statistic.
3. After the table, include a `> **Nota de Margem:**` blockquote with 1-3 sentences of sharp interpretation — "so what?" for this cross. Never repeat what the table already shows. Connect to business implications. Reference qualitative verbatims if available.
4. Zero prose fluff. Go straight to the data. The first text after the heading is the table. No introductory paragraphs, no "Based on the data provided..." throat-clearing.

**Output Format section:**

Show a fenced markdown block with the exact table template. The template MUST include:
- Variable names as placeholders `{VarX}`, `{VarY}`
- N column with `:--:` alignment
- Dynamic VarY category columns (at least 2 shown, with note that more are added for 3+ groups)

- Margin note blockquote after the table
- Every percentage includes MoE notation: `{pct}% (±{moe}%, 95% CI)`
- The table header row contains the key finding (not just "{VarX} × {VarY}")

**Critical constraints to embed:**
- NEVER hardcode a test mapping (e.g., "if categorical then χ²") — the Statistician consults the matrix and exercises judgment (per D-04, D-05)

- Effect size is mandatory — significance without effect size is insufficient

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User data → Statistician stage | Raw variable data enters the statistical pipeline. The Statistician must validate data types and flag impossible values before computing test statistics. |
| Statistician → Critic | Test selection and computed statistics cross the internal agent boundary. The Critic can BLOCK invalid selections — this is a gated boundary, not a pass-through. |
| Critic → Tufte Designer | Validated results and caveats cross into user-facing output. The Designer must not suppress Critic-flagged caveats. |
| Tufte Designer → User | Final output is visible to user. Statistical hallucinations that survive Critic validation become user-visible false claims. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | User-provided VarX/VarY data | mitigate | The Statistician validates data types (categorical vs. continuous) by inspecting the actual data, not trusting variable labels. Flag impossible values (negative counts, %>100%). The Specifies section in the prompt instructs: "Determine data type of VarX and VarY by reading the actual data — do not assume from variable names." |
| T-02-02 | Tampering | Statistical computation (LLM hallucination) | mitigate | The Critic validates test selection against data types and cross-checks test statistics against reported N and % (constitution Article 2 enforcement). Any unreconstructable statistic is flagged. The Critic has BLOCK authority — invalid output cannot reach Tufte Designer. |
| T-02-04 | Repudiation | Missing data / excluded cases | mitigate | The Statistician reports missing data rates per constitution Article 6c. The Tufte Designer includes N in the table so the user can verify counts. Any variable with >10% missing is flagged with bias direction assessment. |
| T-02-05 | Elevation of Privilege | Test assumption bypass | mitigate | The Statistician MUST check assumptions (normality, homogeneity, cell frequencies) before applying parametric tests. The Critic verifies assumptions were checked. If assumptions are violated, the Statistician falls back to non-parametric equivalents — this constraint is in the Execution Steps, not advisory. |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No external packages — meta-prompt project with no package manager installs. |
