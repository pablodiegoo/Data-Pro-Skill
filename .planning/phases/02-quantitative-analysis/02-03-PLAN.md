Insert `/dps-execute` between the `/dps-cross` section's closing `---` divider and the `## Command Reference` header.

**Purpose line:** "Run autonomous multi-cross quantitative analysis. Reads the `/dps-setup` manifesto to identify segments and metrics, derives cross combinations, runs each through the full agent loop (Statistician → Critic → Tufte Designer), and produces a consolidated Tufte report. Works independently from `/dps-plan` (per D-08). If `/dps-plan` output exists, uses it as a starting point but adapts to actual data (per D-09). Use this when you want comprehensive analysis without manually specifying each cross."

**Execution Steps:**

Step 1 — Read the manifesto. Load the YAML frontmatter and segment matrix from the `/dps-setup` output. Extract: all segment names, all tracked metrics, and sample sizes. If no manifesto exists, inform the user: "Execute `/dps-setup` primeiro para definir os segmentos e métricas." Do not proceed without the manifesto — `/dps-execute` requires segments to know what to cross.

Step 2 — Determine crosses to run. If `/dps-plan` output exists in the current context, scan it as a starting point — use its suggested crosses as the initial candidate set. If no plan exists, derive crosses algorithmically: every segment × metric combination from the manifesto. If the number of crosses exceeds 20, prioritize: (a) crosses with largest N per cell, (b) crosses where metrics show high variance between segments in the manifesto. Include a note in the output listing which crosses were prioritized and which were deferred. If ≤20 crosses, run all combinations.

Step 3 — Run each cross through the full agent loop. For EACH cross in the determined set: (a) Statistician selects test and computes statistics (same as `/dps-cross`), (b) Critic validates test selection and constitution enforcement, (c) Tufte Designer formats the cross result as a crosstab table with margin note. If a cross fails validation (Critic blocks it), include it with the Critic's caveat rather than silently dropping it — transparency over completeness.

Step 4 — Consolidate. Combine all crosstab outputs into a single Tufte report with this structure:
- YAML frontmatter: `project`, `date`, `sample_size`, and `crosses_run` (integer count)
- `# {Project Name} — Análise Quantitativa Completa` as the top-level heading
- `> **Nota de Margem:**` overview blockquote summarizing key patterns across all crosses (1-3 sentences)
- Each cross as a `## {VarX} × {VarY} — {Key Finding}` subsection with its table and margin note
- Final `## Síntese dos Achados` section: 3-5 bullet points of the most impactful findings across all crosses, each backed by at least one test statistic

Step 5 — Key findings synthesis. The final synthesis section must: (a) rank findings by effect size, not p-value (a large effect with p=0.06 is more actionable than a tiny effect with p=0.001), (b) note which findings are consistent across multiple crosses (convergent evidence), (c) flag any finding that contradicts another cross (divergent evidence — surface the tension, do not resolve it silently), (d) include at most 5 bullet points — prioritize ruthlessly. The Critic audits the synthesis for overgeneralization and spurious patterns.

**Critical constraints:**
- `/dps-execute` is autonomous — the user does not specify which crosses to run
- `/dps-execute` MUST work without `/dps-plan` per D-08 — the cross derivation in Step 2 covers this
- If `/dps-plan` exists, treat it as a starting point per D-09, but deviate from the plan if the actual data suggests different tests
- Every cross runs the full agent loop — no shortcuts, no skipping Critic validation
- The consolidated report is a SINGLE output, not incremental — all crosses run before any output is shown
- If the manifesto has only 1 segment or 1 metric, output "Manifesto contém apenas {1 segmento / 1 métrica} — análise multi-cruzamento requer pelo menos 2 segmentos e 1 métrica (ou 1 segmento e 2 métricas). Execute `/dps-cross` para cruzamentos individuais."

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `/dps-setup` manifesto → `/dps-execute` cross derivation | The manifesto's segment definitions are trusted as the source of truth. If the manifesto contains errors (wrong N, mislabeled segments), all derived crosses inherit those errors. |
| `/dps-plan` output → `/dps-execute` starting point | The plan is a suggestion. `/dps-execute` must not blindly follow a plan that contradicts actual data — this boundary is where D-09 applies ("adapts to actual data"). |
| Multi-cross Statistician → Critic | Each cross runs the full agent loop independently. One invalid cross must not contaminate other crosses' results. The Critic validates per-cross, not globally. |
| Synthesis section → User | The "Síntese dos Achados" section aggregates findings across crosses. Overgeneralization risk is highest here — the Critic audits the synthesis specifically for cross-cross spurious patterns. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | Manifesto-derived cross combinations | mitigate | The `/dps-execute` prompt instructs extracting segments and metrics from the YAML frontmatter — a structured format that is harder to misparse than free text. If the frontmatter is missing or malformed, the command halts with an explicit error message. |
| T-02-02 | Tampering | Autonomous cross selection (which crosses to run) | mitigate | Cross derivation is algorithmic (segment × metric from manifesto). No LLM judgment in which crosses to run — deterministic pairing. Only when >20 crosses is there a prioritization heuristic, and the output MUST report which crosses were prioritized and which deferred — full transparency. |
| T-02-03 | Tampering | Synthesis overgeneralization | mitigate | The synthesis section is audited by the Critic before delivery. Critic checks: (a) every bullet point backed by at least one test statistic, (b) no finding claimed as "significant" without p < 0.05, (c) convergent evidence distinguished from single-cross findings, (d) divergent evidence surfaced, not silenced. The synthesis is limited to 5 bullet points maximum — this constraint reduces hallucination surface area. |
| T-02-04 | Repudiation | Cross failures (Critic blocks) | mitigate | If the Critic blocks a cross, `/dps-execute` includes it with the Critic's caveat rather than silently dropping it. The user sees all crosses — both passing and blocked. This prevents selective reporting bias where only "good" results survive. |
| T-02-05 | Elevation of Privilege | `/dps-plan` as unauthorized constraint | mitigate | Per D-09, `/dps-execute` uses the plan as a "starting point" but adapts to actual data. The prompt explicitly instructs: "If the Statistician determines a different test is appropriate based on actual data characteristics, use that test — do not force the plan's recommendation." The plan is advisory, not authoritative. |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No external packages — meta-prompt project with no package manager installs. |
</threat_model>
